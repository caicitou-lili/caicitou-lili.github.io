Claude Code 动态工作流 — 为每个任务定制专属引擎

上周，Claude Code 发布了动态工作流（Dynamic Workflows）功能。Claude 现在可以在运行中动态编写自己的编排脚本，为手头的任务定制专属的执行引擎。

虽然默认的 Claude Code 是为编程构建的，但事实证明很多任务都类似于编程任务。然而，有些特定类型的任务需要在 Claude Code 之上构建定制引擎才能达到最佳效果——比如深度研究、安全分析、Agent 团队协作、代码审查。

Workflows 让你可以动态创建这些引擎，让 Claude 原生解决以上所有问题，还能在团队间分享和复用。

📌 原文发表于 Claude 官方博客，作者 Thariq Shihipar（Anthropic 技术团队成员）。本文为中文整理版。

示例 Prompt：感受一下 Workflows 能做什么

在深入技术细节之前，先看几个例子：

  • “这个测试大概每 50 次有 1 次失败。建一个工作流来复现它、提出假设并在 worktree 中进行对抗性测试，目标是不找到一个有效理论就不要停。”
  • “用一个工作流，翻我最近 50 个会话，挖出我反复被纠正的地方，把重复出现的写成 CLAUDE.md 规则。”
  • “拿我的商业计划书，跑一个工作流，让不同的 Agent 从投资人视角、客户视角、竞争对手视角去挑刺。”
  • “这有个文件夹，里面 80 份简历，用工作流给后端岗排序，对前 10 名做双重复核。”
  • “帮我写一个工作流，生成一堆名字方案然后跑一场锦标赛，选出前三。”
  • “把我博文草稿里的每一个技术论断都对照代码库验证一遍，我不允许有错。”

动态工作流如何工作

动态工作流执行一个 JavaScript 文件，其中包含几个用于生成和协调子 Agent 的特殊函数：

  • agent() — 衍生一个子 Agent，可以指定 schema 获得结构化输出
  • parallel() — 并发执行多个任务（屏障：等待全部完成）
  • pipeline() — 流水线模式：每个 item 独立跑完所有阶段，无屏障
  • phase() — 启动一个新阶段，后续 agent 调用归入该阶段
  • log() — 输出进度消息

工作流还支持标准 JS 函数（JSON、Math、Array），并且可以指定每个子 Agent 使用的模型以及是否在隔离 worktree 中运行。

如果工作流被中断（用户操作或退出终端），恢复会话后工作流会从断点继续。

为什么需要动态工作流？

当你让默认的 Claude Code 处理一个任务时，它需要在同一个上下文窗口中规划和执行。对于很多编程任务来说这很高效，但在长周期、大规模并行和高度结构化的对抗性任务中会出现几种失败模式：

Agent 惰性（Agentic Laziness）：Claude 在完成复杂多步骤任务的部分进展后就宣告完成，比如安全审查只处理了 50 项中的 20 项。

自我偏好偏差（Self-Preferential Bias）：Claude 倾向于偏好自己的结果，尤其在需要按照评分标准验证或评判时。

目标漂移（Goal Drift）：经过多次轮次，尤其在上下文压缩后，对原始目标的保真度逐渐丧失，边界条件、”不要做 X”等约束会被丢失。

工作流通过编排各自拥有独立上下文窗口和聚焦目标的多个 Claude，来对抗这些失败模式。

动态 vs 静态工作流

你之前可能用 Claude Agent SDK 或 claude -p 创建过静态工作流来协调多个 Claude Code 实例。

但静态工作流需要覆盖所有边缘情况，所以通常比较通用。有了 Claude Opus 4.8 和动态工作流，Claude 现在足够聪明，可以为你的具体用例量身编写定制引擎。

实用模式

你可以直接让 Claude 建一个工作流，或使用触发词 “ultracode” 来确保 Claude Code 创建工作流。以下是 Claude 在构建工作流时常组合使用的模式：

分类-执行（Classify-and-Act）

用分类 Agent 判断任务类型，然后路由到不同 Agent 处理。或最后用分类器判断输出。

扇出-合成（Fan-out-and-Synthesize）

把任务拆成多个小步骤，每个步骤跑一个 Agent，最后合成结果。合成步骤是屏障，等所有扇出 Agent 完成后合并。

对抗性验证（Adversarial Verification）

对每个 Agent 的输出，衍生独立的验证 Agent 进行对抗性审核。

生成-筛选（Generate-and-Filter）

生成一批想法，按评分标准筛选，去重，只保留经过验证的高质量结果。

锦标赛（Tournament）

让多个 Agent 竞争同一个任务，使用不同方法，然后由评判 Agent 两两比较，直到选出胜者。

循环至完成（Loop Until Done）

对于工作总量未知的任务，持续衍生 Agent 直到满足停止条件（无新发现、无更多错误）。

典型用例

迁移和重构

Bun 用工作流完成了从 Zig 到 Rust 的重写。关键是把任务拆成操作步骤（调用点、失败测试、模块等），每个修复在独立 worktree 中完成，再经过对抗性审查后合并。

深度研究

Claude Code 内置的 /deep-research 技能就是用动态工作流实现的。不仅仅用于网页搜索，还可以用来从 Slack 中整理状态报告，或深入探索代码库。

深度验证

如果一个报告需要核实每一个事实论断，可以让一个 Agent 识别所有事实，然后为每一项衍生独立的验证子 Agent，甚至可以再衍生验证 Agent 来审核来源质量。

排序

对于需要定性判断的排序任务（如按 Bug 严重程度排序支持工单），用锦标赛模式、两两比较流水线来避免单次 prompt 质量下降。

记忆与规则遵守

创建验证 Agent（每规则一个），对抗性检查规则是否被违反。反过来也可以：挖掘最近的会话历史，聚类重复修正，对抗性验证每个候选规则，精炼后写回 CLAUDE.md。

根因调查

从不相交的证据来源（日志、文件、数据）衍生假设生成 Agent，每个假设再经过验证和反驳 Agent 的审核。不仅适用于代码——销售下降原因、数据管道故障等都可以用。

大规模分类

每个团队都有处理不完的支持队列/Bug 报告。分类工作流对每个项目分类、去重，然后尝试修复或升级给人类。搭配 /loop 可让 Claude 持续运行。

探索与品味

当方案选择涉及品味（如设计、命名），让 Claude 探索多种方案，审查 Agent 按评分标准评判，完成条件是达到标准。也可用锦标赛模式选出最优方案。

评估（Evals）

在独立 worktree 中运行轻量评估，再衍生比较 Agent 对照标准评分，比如评估和精炼一个你创建的 Skill。

模型与智能路由

创建针对你任务的分类 Agent，预先研究确定最佳模型——比如先探索模块的复杂度，再决定用 Sonnet 还是 Opus。

什么时候不应该用工作流

工作流还很新。虽然很多场景效果出众，但并非每个任务都需要，可能会消耗更多 Token。

对常规编程任务，问自己：真的需要更多算力吗？ 比如传统的编程任务通常不需要 5 个审查者的评审团。

构建技巧

详细 Prompt：用上述具体技术描述的提示词，能产生最佳结果。

快速工作流：工作流不只适用于大型任务。你也可以让模型用”快速工作流”——比如对某个假设做一次快速对抗性审查。

搭配 /goal 和 /loop：可重复的工作流（分类、研究、验证），配合 /loop 定时运行，配合 /goal 设定硬性完成要求。

Token 预算：可以设定显式的 Token 使用预算，如”用 10k token”。

保存和分享：在工作流菜单按 “s” 保存，文件存入 ~/.claude/workflows 或通过 Skill 分发。把 JS 工作流文件放入 Skill 文件夹并在 SKILL.md 中引用即可。

总结

工作流是扩展 Claude Code 的一种全新方式。把它当作一个起点——如何使用它们的最佳实践还在被不断发现。

真正让人兴奋的是：Claude 不再只是一个执行单步任务的工具，而是一个能为自己定制执行引擎的元工具——为每个任务创造最合适的协作模式。
