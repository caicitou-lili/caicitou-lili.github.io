别再伺候 AI 了：5 个方法狠狠压榨 Codex

这篇文章想解决一个很具体的问题：

你已经开始用 Codex 了，但每次还是要自己复制资料、补充背景、反复解释需求，最后感觉不是 AI 在帮你省时间，而是你在陪 AI 加班。

我自己也经历过这个阶段。Codex 真正好用，不是因为我写出了多么神的 prompt，而是我开始把一些重复动作交给它自己跑。

这篇就把我目前最常用的 5 个方法整理出来。你不需要一次全配完，先挑一个最适合自己的场景试起来就行。

1. 插件：先让 Codex 长出手脚

没有插件的 Codex，只能回答你。接上插件以后，它才开始能做事。

我最直观的感受来自 Gmail。以前处理邮件很烦——先扫标题、再点进去看、再判断要不要回、再想怎么回。现在直接让 Codex 用 Gmail 插件处理第一轮：哪些紧急、哪些要回、哪些只是通知，先筛出来。需要回复的，它先起草。我只负责看一眼，改两句。

Gmail、Slack、Figma、Notion、Google Drive、GitHub，这些插件接上以后，Codex 就不再只是一个聊天 AI，而是能进工具里帮你干活的助手。

小白配置：

  1. 打开 Codex app
  2. 进左侧或设置里的 Plugins
  3. 搜索 Gmail / Google Drive / GitHub 等插件
  4. 点安装或连接，按提示登录授权
  5. 回到对话里，明确告诉 Codex 用哪个插件

2. Goal Mode：不要给任务，给目标

很多人用 Codex 是这样用的：”帮我看一下这个文件。””再改一下。””这里不对。””继续。”

这其实还是你在牵着它走。你累，它也容易跑偏。

Goal Mode 更像是你给它一个终点，然后让它自己拆步骤、执行、检查、修正。

OpenAI 在 2026 年 5 月的 release notes 里提到，Goal mode 已在 Codex app、IDE extension 和 CLI 里可用。你可以定义结果和成功标准，让 Codex 持续朝目标推进。

举个例子——改论文格式。我以前会一步步说：先改标题，再调引用，再检查格式。后来发现没必要。更好的方式是直接告诉它：

“把这篇论文改到目标期刊格式，成功标准是标题、摘要、引用、图表编号、参考文献都符合要求，最后给我修改总结。”

普通模式是你说一步，它走一步。Goal Mode 是你定义终点，它自己找路。

3. 自动化：让 Codex 每天自己上班

真正省时间的东西，不是”我叫它，它很快”。是**”我不叫它，它也会按时出现”**——这就是 Automations。

别一上来就让它”帮我赚钱””帮我运营账号”，太虚了。先从每天、每周重复的小流程开始。比如：

  • 每天早上检查项目错误日志，整理新增问题
  • 每天早上整理 AI / Codex / MCP 的重要更新

我自己比较喜欢后者。醒来之后先看一份筛过的简报，而不是钻进信息流里捞半小时。

小白配置：

  1. 先在普通对话里手动跑一次
  2. 确认输出格式满意
  3. 对 Codex 说：把刚才这个任务创建成自动化
  4. 去左侧 Automations 检查任务
  5. 确认频率、提示词、状态
  6. 先跑 2-3 次，再改 prompt

4. MCP + Obsidian：把 Codex 接进你的知识库

互联网给 Codex 外部信息。Obsidian 给 Codex 你的个人上下文。 这俩不一样。

很多人说 AI 不懂自己。其实也正常——你的笔记、经验、素材、项目复盘都在 Obsidian 里，Codex 没看见，当然只能泛泛地写。

MCP 解决的就是这个问题。通过 MCP，你可以让 Codex 连接 Obsidian、文档库、内部工具。

小白配置：

  1. 在 Obsidian 安装 Local REST API with MCP
  2. 打开插件，确认 REST API 服务已启动
  3. 记录插件里的 API Key
  4. 在 Codex 的 MCP 配置里添加 Obsidian server
  5. 重启 Codex
  6. 让 Codex 测试读取一篇笔记

这一步打通的不是一个工具，而是你的长期积累。

5. Skills：把常用流程固化

如果一个流程你重复做了 3 次，就应该把它变成 Skill。

插件解决”去哪里拿资料”，Skills 解决”按什么流程做”。Skill 就是你写给 Codex 的 SOP。

比如你经常让 Codex 写 X 长文——不要每次都重新说：开头要有钩子、不要 AI 味、先讲痛点、再给方法、每个方法要有例子、最后要有行动号召。这些东西，应该沉淀成一个 Skill。

小白配置：

  1. 先让 Codex 跑通一次完整任务
  2. 结果满意后，让 Codex 把流程整理成 Skill
  3. 检查 Skill 是否包含适用场景、输入要求、流程、质量标准
  4. 下次直接调用

更高级一点，可以做父子 Skill：父 Skill 判断任务类型（长文/表格/查资料/报告），子 Skill 负责具体执行。

最后

很多时候不是你不会用 AI，而是我们还停留在”每次都重新解释一遍”的阶段。

更推荐的方式：先从一个最小场景开始。 比如先装一个 Gmail 插件、或者先把每天都要看的信息做成自动化、或者先把你最常写的一类文章做成 Skill。

不用一下子把 5 个方法全部配齐。你只要先跑通一个，就会明显感觉到——Codex 不再只是回答你，而是开始替你分担一小段流程。
